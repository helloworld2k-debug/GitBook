# 语言策略决策建议 - 2026-05-28

## 背景

当前线上语言为：

- English: `en`
- 繁体中文: `zh-Hant`
- Japanese: `ja`
- Korean: `ko`

当前保留的未合并分支 `language-simplified-zh-admin-en-zh` 试图把语言范围收敛为：

- English: `en`
- 简体中文: `zh`

该分支会删除 `messages/ja.json`、`messages/ko.json`、`messages/zh-Hant.json`，新增 `messages/zh.json`，并更新大量测试。

## 关键判断

不建议直接合并旧分支。

原因：

- 它是产品策略变更，不是技术清理。
- 它会移除日语、韩语、繁体中文三个现有公开语言入口。
- 它基于旧代码，当前会误删 `src/config/site.ts` 里的 `one_day` 捐助 tier。
- 它一次改动 36 个文件，回归面偏大，不适合和后台 UX 收尾混在一起。

## 方案比较

### 方案 A：维持当前四语言

保留 `en / zh-Hant / ja / ko`。

优点：

- 风险最低。
- 不影响现有 URL、SEO、语言入口和测试覆盖。
- 不需要迁移或重定向。

缺点：

- 后续新增文案要继续维护 4 套翻译。
- 如果日语/韩语质量不足，可能影响用户信任。

适合情况：

- 当前国际化入口已经上线并希望保持稳定。
- 暂时没有足够时间做语言策略和翻译质量治理。

### 方案 B：收敛为英文 + 简体中文

把 active locales 改为 `en / zh`，移除 `zh-Hant / ja / ko`。

优点：

- 维护成本最低。
- 更符合中文主导开发和运营节奏。
- 文案质量更容易统一。

缺点：

- 会移除已上线的繁中、日语、韩语路径。
- 需要明确旧路径处理策略，例如 `/zh-Hant`、`/ja`、`/ko` 是否 301/302 到 `/zh` 或 `/en`。
- 需要重做语言切换器、静态路由、测试和消息完整性检查。

适合情况：

- 明确决定短期只服务中英用户。
- 愿意接受旧语言 URL 的重定向和潜在 SEO 波动。

### 方案 C：新增简体中文，暂时保留繁中/日/韩

把 active locales 扩展为 `en / zh / zh-Hant / ja / ko`，先新增简中，不删除旧语言。

优点：

- 用户影响最小。
- 可以先验证简中体验和转化，再决定是否下线旧语言。
- 避免一次性删除大量消息文件。

缺点：

- 维护成本最高。
- 短期要维护 5 套消息。
- 如果没有翻译流程，新增语言会放大文案同步成本。

适合情况：

- 想要简体中文，但不想立刻砍掉现有语言入口。
- 希望用数据或运营观察再决定是否收敛。

## 推荐结论

推荐采用分两步的保守路线：

1. 短期选择方案 A：维持当前四语言，不合并旧分支。
2. 如果确实需要简体中文，再新开分支执行方案 C 的最小版本：新增 `zh`，暂不删除 `zh-Hant / ja / ko`。

这样可以避免把“新增简中”和“删除现有语言”绑成一个高风险发布。

## 建议执行顺序

如果要继续推进语言策略，建议按下面顺序：

1. 新建干净分支，例如 `feature/simplified-chinese-locale`.
2. 从当前 `main` 新增 `messages/zh.json`，不要复用旧分支的整体提交。
3. 保留 `one_day` 捐助 tier 和当前所有业务改动。
4. 先让语言切换器显示 `English / 简体中文 / 繁體中文 / 日本語 / 한국어`。
5. 跑完整消息 key parity 测试，确认 `en`、`zh`、`zh-Hant`、`ja`、`ko` key 一致。
6. 做 `/zh` 首页、版本页、支持页、后台入口的浏览器 QA。
7. 单独评估是否下线 `ja / ko / zh-Hant`，不要和新增 `zh` 同一个 PR 完成。

## 暂不建议做的事

- 不建议直接 merge `language-simplified-zh-admin-en-zh`。
- 不建议现在删除 `ja / ko / zh-Hant`。
- 不建议把语言策略变更混入后台 UX 或运营巡检提交。
- 不建议在没有重定向方案前移除旧语言 URL。

## 当前保留项

保留本地和远端分支：

- `language-simplified-zh-admin-en-zh`

用途：

- 作为历史参考。
- 不作为直接合并对象。
